# Async Processing

Async processing adalah pola pemrograman di mana sebuah proses tidak dieksekusi secara blocking sampai selesai, melainkan dijalankan secara non-blocking sehingga aplikasi bisa tetap melanjutkan pekerjaan lain. Biasanya, hasil dari proses async akan dikembalikan melalui mekanisme seperti callback, promise/future, goroutine + channel (di Go), atau event handler. Pendekatan ini sangat berguna untuk meningkatkan throughput dan responsivitas sistem, terutama pada operasi I/O-heavy seperti akses database, HTTP request, pengiriman email, atau pemrosesan data yang memakan waktu. Dengan async processing, beban kerja bisa dipindahkan ke background worker sehingga request utama tetap cepat selesai.

Dalam implementasinya, async processing bisa dibangun menggunakan message queue atau message broker (event bus), dan keduanya punya tujuan yang sedikit berbeda. Message queue lebih cocok untuk pola “task distribution” (command-based), di mana satu pekerjaan harus dikerjakan sekali oleh satu worker saja, misalnya: kirim email, generate report, atau proses pembayaran—contohnya memakai RabbitMQ atau Redis Queue. Sementara message broker / event bus lebih cocok untuk “event distribution”, di mana satu event bisa dikonsumsi banyak service secara independen, misalnya OrderCreated yang memicu inventory, analytics, dan notification service sekaligus—contohnya Kafka. Di sini penting membedakan domain event vs command: command adalah instruksi eksplisit untuk melakukan sesuatu (“ChargePayment”, “CreateOrder”) dan biasanya memiliki satu eksekutor utama; sedangkan domain event adalah fakta yang sudah terjadi (“OrderCreated”, “PaymentSucceeded”) dan bisa dikonsumsi banyak pihak tanpa mengubah sumber event. Kesimpulannya, gunakan message queue ketika butuh satu tugas dikerjakan satu kali dengan ownership jelas, dan gunakan event bus ketika ingin menyebarkan fakta bisnis antar service secara loose-coupled dan scalable.
